TASK-001: Add dot-notation config key resolution helper to common/clicfg
Date: 2026-05-02T00:00:00Z
================================================================================

Summary:
Added `ResolveConfigKey(key string, cfg *Config) (ConfigScope, string, error)` to
common/clicfg/clicfg.go. The function resolves a dot-notation config key string to
a namespace (GlobalScope or AuraScope) and a bare key name. Keys prefixed with "aura."
resolve to AuraScope with the prefix stripped; all others resolve to GlobalScope.
Global-only keys (e.g. "output") cannot be addressed via the "aura." prefix.
Unrecognised keys in either namespace return a clierr.UsageError. Table-driven tests
covering all acceptance criteria were added to common/clicfg/clicfg_test.go.

Files Changed:
- common/clicfg/clicfg.go (modified: added ResolveConfigKey function at end of file)
- common/clicfg/clicfg_test.go (modified: added TestResolveConfigKey table-driven test, added require import)

Key Decisions:
- Function signature: ResolveConfigKey(key string, cfg *Config) (ConfigScope, string, error)
  takes the full Config so it can check both Global.ValidConfigKeys and Aura.ValidConfigKeys
- Global-only key collision check: if "aura.X" is requested and X is in Global.ValidConfigKeys,
  return a specific error message noting the key is global-only, not just "unrecognised"
- Used clierr.NewUsageError for all error cases, consistent with rest of codebase
- Used strings.HasPrefix / strings.TrimPrefix for "aura." prefix handling

  (safe since cfg is fully initialized before commands are built)

  "aura.default-tenant" rather than just "default-tenant"

  giving the standard cobra "invalid argument" error message

Summary:
Updated `neo4j-cli/internal/subcommands/config/set.go` to accept both global keys
(e.g. "output") and aura-prefixed dot-notation keys (e.g. "aura.default-tenant").
The command now uses `clicfg.ResolveConfigKey` for key validation in Args and for

- Removed `clierr` import from set.go since error generation is now fully delegated to `ResolveConfigKey`
- Error message for unknown keys changed from "invalid config key specified: X" to the resolver's `"invalid config key: \"X\""` format; tests were updated to match

Date: 2026-05-02T04:00:00Z

Summary:
Fixed a duplication bug in `Config.Printable()` where "output" appeared twice in
the AuraScope list output. For AuraScope, global keys were already added first
(from `cfg.Global.ValidConfigKeys`), but the aura keys loop also included "output"
because it was appended to `validAuraConfigKeys` at config creation time. Added a
`continue` guard to skip aura keys that are already present in global keys. Added
a new test `TestListConfigOutputAppearsOnce` in `list_test.go` that verifies
"output" appears exactly once and that aura keys appear without prefix.

Files Changed:
- common/clicfg/clicfg.go (modified: added deduplication guard in Printable() for AuraScope)
- neo4j-cli/aura/internal/subcommands/config/list_test.go (modified: added TestListConfigOutputAppearsOnce test)

Key Decisions:
- Fix lives in `Config.Printable()` rather than in `list.go` or `set.go`; the
  `get.go` and `set.go` ValidArgs (`cfg.Aura.ValidConfigKeys`) were already correct
  since global keys are appended once at config creation, not duplicated
- The table output duplication was masked in JSON rendering (map assignment
  overwrites the duplicate key silently) but would show double rows in table output
- Used `strings.Count` in the test to assert exactly one occurrence of "output"
